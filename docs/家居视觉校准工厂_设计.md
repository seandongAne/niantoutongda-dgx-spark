# 家居视觉校准工厂 — 产品方向与工程设计 v1

> 2026-07-16(D4 夜)Sean 拍板的产品方向升级。触发:GT 判卷证明效果对场景
> 杂乱度高度敏感(儿童房 vs 整齐房间),而"开发者看着视频手调 GDINO"不是
> 可交付的产品形态。本文冻结方向与护栏;赛程内只做 §6 的最小切片。

## 0. 一句话

不让每个用户重新调 GDINO——开发期用阶跃双模型持续运转"家居视觉校准工厂",
把成熟结果冻结成**版本化能力包(household-pack-vN)**;运行时纯本地加载 pack,
用户侧是"自动盘点 + 轻确认",不暴露任何 threshold/prompt 概念。

## 1. 为什么这是对的(证据链)

- **词表+工作点是该检测器的第一杠杆**:v5→v6 召回 0.464→0.710(+24.6pp)
  全部来自词表与推理路径改进,零训练。值得工厂化的正是这个已被证明的杠杆。
- **逐场景手调不可产品化**:GT 判卷显示杂乱儿童房碎轨率 0.206、FP/帧 17.4,
  同一套配置在整齐房间会好得多——说明工作点本质上是"数据分布的函数",
  应该在开发期对分布做覆盖,而不是在运行期让用户承担校准。
- **一半设计已存在**:S2.5-6 视觉教师+数据工厂、playbook 用途②③、
  S2.5-3 prompt_search 打分器、S2.5-5 hardval 评测。48 个候选短语待打分
  就是工厂 v0 的第一炉——本设计是把已有件重组为持续闭环,不是新起炉灶。

## 2. 工厂闭环(dev-time,云用途)

```
公共/授权真实家居图片(不含任何家庭素材)
        ↓
Step 3.7 Flash(视觉教师)
  类别体系、英文同义词/搜索词、易混物品组、失败归因
        ↓
Step Image Edit 2(困难版本工厂)
  光照 / 距离尺度 / 遮挡 / 杂乱背景 / 模糊 等定向增强
        ↓
GDINO 自动跑多组 prompt × 工作点(prompt_search 网格)
        ↓
真实留出视频(dev_a hardval + dev_b)三指标裁决
        ↓
发布 household-pack-v1、v2……(冻结、带 manifest)
```

分工:教师负责**广度**(词表体系与易混结构),编辑负责**难度**(把易样本
变难样本用于排序压测),真实留出集负责**真相**(唯一发布门)。

## 3. 四条工程护栏(没有这些,工厂会自欺)

1. **合成图只排序,真实集才发布。** 网络图/生成图与手机视频帧存在分布差;
   Image Edit 产物只用于给候选词与工作点**排序**和压力测试,任何 pack 的
   发布门 = 真实留出视频上锚点召回/碎轨率/FP-每帧三指标不劣于上一版。
2. **保身份校验。** 困难版本必须过标签保持检查(编辑前后主体可检出且
   同类、框 IoU 下限;每批人工抽检 ≥10%,通过率入 manifest)。编辑把物体
   改没了还当正样本,是这类数据工厂最常见的自毁模式(承 S2.5-6 既有条款)。
3. **能力包只管检测层。** pack 校准的是 S2:词表、per-class 工作点、易混组、
   NMS 别名组。S3 跨视频身份的嵌入可分性天花板(同锚 p50 0.673 vs 跨锚
   0.534)它救不了——属性接线仍是独立主战场,两者不得互相冒充战果。
4. **红线继承且更干净。** 工厂全部 dev-time;云上只喂公共/授权图片,
   **家庭素材在此设计下彻底不出境**(只在本地当留出集);云输出只是候选
   (playbook 红线 3);每炉 token 记账;演示主链只加载冻结 pack,不接云。

## 4. household-pack-vN 规格

| 内容物 | 说明 |
|---|---|
| `vocab.json` | canonical_id / display_label_zh / detection_prompts[] / confusable_group(承 S2.5-1 schema) |
| `working_points.json` | per-class threshold、tile 触发条件、批次分组 |
| `synonyms.json` | 长尾同义词扩展表——运行时**离线查表**,未收录类别自动展开,不调云 |
| `eval_report.json` | 真实留出集三指标 + GT 判卷结果(发布依据) |
| `MANIFEST.json` | 来源图片 hash、教师/编辑模型与 prompt 版本、候选/接受/拒绝数、人工抽检通过率、token 用量、发布判定 |

pack 不可变、语义化版本;回滚 = 换 pack 版本号,与 reid config 冻结纪律同构。

## 5. 运行时形态:自动盘点 + 轻确认

1. 上传视频抽少量 contact sheet(复用 S2 关键帧);
2. 本地视觉模型(S5 Nemotron/vLLM)自动提出物品候选("床、台灯、收纳盒、粉色旅行箱");
3. 候选映射到 pack 全局词表;长尾物品经 `synonyms.json` 自动展开;
4. GDINO 检测 + 追踪(全部本地,加载冻结 pack);
5. 只对真正不确定的少数物品请用户点一次确认(复用 S3 澄清机制与互选封顶)。

用户不需要预先列全物品清单,也不接触 threshold/prompt/GDINO 概念。
产品声明升级为:**"对常见家居物品自动盘点,并能通过自动长尾词表扩展处理
未收录类别;用户只处理少量不确定项。"**——比"必须预先声明 15 件物品"宽得多,
且每个字都有留出集数字背书,诚实。

注意:5 步里 1/4/5 已存在,2 与 S5 属性接线共用同一套 vLLM 服务(顺路),
3 只是查表——运行时侧的增量工程量很小,大头全在开发期工厂。

**命名权威原则(2026-07-16 目检定案)**:用户可见的物品名称一律以
S5 本地 VLM 读 hero 图的输出为准;GDINO 的 raw_label 只是召回内部键,
永不直接展示。依据:71 对新增链接目检显示非锚点轨的箱体类误名高发
(mini_fridge/wardrobe 实为玩具箱等)且存在跨 prompt 拼接畸形标签
("baby monitor storage box"),但 161 条 GT 锚点轨类别 0 错配、
71 对两端类别 0 不一致——误名不破坏链接,只污染展示层,
修复归属 S5 属性接线战役,不另开工。

## 7. dev_b 演示级素材计划(同日拍板)

**定位**:dev_a 不是失败数据集,是被动获得的压力测试集(161 轨 GT 已投入),
保留为 hardval;dev_b 是演示级 + 工厂第二留出集。

**拍摄纪律**:慢速平移;每件物品停留 1–2 秒;先远景一圈再近景一圈;
物品之间留空隙;白天顺光。**顺录口述旁白**("这个蓝色水杯是妹妹的,
搬去新家厨房")。

**一份素材三个用途**:
1. 演示叙事:"整齐房间高表现 + 杂乱儿童房优雅降级、主动提澄清"——
   双数据集对照本身就是路演素材;
2. 工厂发布门第二留出集(§3 护栏 1);
3. A1 旁白协议真实噪声复测素材(赛程内自录,符合"不用家庭历史音频"红线,
   承 `results/stepfun/a1_warmup/PROTOCOL.md` 待办)。

**排期**:穿插在 S5 长任务跑批空档,不占主战场工时。
