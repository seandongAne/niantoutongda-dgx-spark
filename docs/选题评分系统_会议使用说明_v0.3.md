# 选题评分系统 v0.3 · 会议操作手册（书记版）

> 配套文件：`DGX_Spark_选题评分系统_v0.3.xlsx`
> 用途：在一次选题会（约 45–60 分钟）里，用同一套公开标准选出"最适合参赛"的项目——而不是最会讲故事、技术最炫或愿景最大的项目。
> 本手册面向会议书记：你负责操作表格、记录结论，主持人负责控场。

---

## 0. 开会前 10 分钟：准备工作

1. 用 Excel / WPS / LibreOffice 打开文件（文件自带计算结果，打开即可用）。
2. **清空演示数据（必做）**。P01–P03 三行是演示样例，不清掉会污染真实排名：
   - "项目清单"前三行的 A–P 列内容，覆盖成你们自己的候选项目；
   - "会议评分"前三行的 C–N 分数、B 列状态（改回"未评"）、W/X 两列文字，全部清空；
   - "证据索引"第 5–40 行的"证据状态 / 证据摘要 / 来源 / 负责人"清空。
3. 只在**黄色输入格**里打字；蓝灰色是公式列（排名、总分、建议等），**碰坏了整表失灵**。
4. 权重、锚点、阈值都在"参数与锚点"表里，公开可查。会前若调整过权重，确认 D17 合计仍是 **100**。

## 1. 表格结构一览

| 工作表 | 干什么 | 你要填的 |
|---|---|---|
| 使用说明 | 规则速查 | 无 |
| 参数与锚点 | 权重、评分锚点、决策阈值 | 一般不动 |
| 项目清单 | 每个候选项目一行：基本信息 + 6 个硬门槛 | A–P 列、AD 列（下一步实验） |
| 会议评分 | C1–C12 共识分，自动算总分 | B–N 列、W/X 两列 |
| 证据索引 | 每项评分对应一行证据 | D–H 列 |
| 决策面板 | 自动排名 + 建议，只读 | 无 |

## 2. 会议流程（按顺序执行）

### 第 1 步 · 项目陈述 + 填写项目清单

每个项目所有者用同一模板陈述，你同步把要点填入"项目清单"A–J 列：
一句话场景、目标用户、付费方、商业化路径、数据来源、可验证异常、90 秒闭环、Spark 不可替代性。

### 第 2 步 · 过 6 个硬门槛（K–P 列，下拉选择）

| 门槛 | 通过标准 |
|---|---|
| G1 数据 48h 可用 | 合法、可下载、可留离线副本 |
| G2 异常 48h 可定义 | 标签 / 规则 / 代理真值可写成测试 |
| G3 前 72h 最小链路 | 数据输入→模型判断→可见结果能跑通（不是做完产品） |
| G4 核心闭环 ≤90 秒 | 固定回放 90 秒内看完异常→判断→行动→结果（不是路演只有 90 秒） |
| G5 Spark 技术路径 | aarch64、模型、内存、运行时有方案 |
| G6 安全与责任边界 | 隐私、误报、人工确认、禁止动作明确 |

规则：**任一"失败"→ 项目淘汰**；任一"待验证"→ 最好结局只能是"进入 24h 验证"，不能直接当选。
选了"待验证"的，**当场**在 AD 列写清 24 小时内能做完的实验和负责人，例如："下载 100 条 AIS 轨迹，验证能否稳定识别禁区进入事件——张三"。

### 第 3 步 · 会议共识评分（C1–C12，每项 0–5 一个整数）

推荐纪律（防止最会说话的人带节奏）：
1. 主持人朗读该项的关键问题和 1/3/5 分锚点（见"参数与锚点"表）；
2. **每人先在纸上默写自己的分数**；
3. 只对差距 ≥2 分的项展开讨论；
4. 讨论后你记录**一个共识分**——不取平均、不留个人分。

评分语言：

| 分 | 含义 | 证据要求 |
|---|---|---|
| 0 | 没有信息 / 不成立 | — |
| 1 | 主要是口号 | 仅假设 |
| 2 | 部分成立，关键环节缺失 | 少量来源 |
| 3 | 可信最低线：路径清楚可落地 | 有来源或可复现说明 |
| 4 | 已有原型 / 数据 / 测量 | **必须有来源** |
| 5 | 已验证、可复现、可演示 | **必须已验证** |

提醒：不确定 ≠ 3 分；"不确定"就打低分并写进保留意见。

每个项目评完后填三样：
- **B 列 会议状态**：未评 / 评审中 / 已达成共识（不选"已达成共识"，系统不出结论）；
- **W 列 共识依据**：为什么是这个分数；
- **X 列 保留意见**：没谈拢的分歧、待验证的假设——这些就是下一轮实验清单。

### 第 4 步 · 登记证据（证据索引表）

每个项目 × 每个评分项一行，D 列下拉选状态，E–H 填摘要、来源路径、负责人、复验日期。

| 证据状态 | 折算 |
|---|---|
| 未提供 | 0% |
| 假设 | 25% |
| 有来源 | 60% |
| 已验证 | 100% |

**v0.3 硬规则**：某项打了 4–5 分，但证据状态低于"有来源"，系统**自动按 3 分参与计算**。
"会议评分"表 AK 列显示每个项目被封顶了几项——AK 不为 0，说明有高分没证据撑腰，当场补登证据或接受降分。

### 第 5 步 · 看决策面板宣读结论

判读顺序：**门槛 → 总分 → 证据强度 → 评分完整度**。

| 结论 | 触发条件 |
|---|---|
| 优先 | 门槛全通过 + 总分 ≥80 + 证据 ≥70% |
| 进入 24h 验证 | 门槛待验证且总分 ≥60；或门槛通过、总分 ≥70 但没到优先线（**高分低证据走这里**） |
| 备选 | 总分 60–69 |
| 放弃 | 总分 <60（门槛待验证但总分 <60 的也直接放弃） |
| 淘汰 | 任一硬门槛失败 |
| 会议待完成 | 12 项没评完，或会议状态没确认"已达成共识" |

两条铁律：
- 90 分的项目，只要数据或 Spark 路径还是"待验证"，也**不能直接当选**；
- 高分低证据不是冠军，是"最值得优先花 24 小时验证的假设"。

已知小毛病：被淘汰的项目仍会按总分占据排名位（面板上会同时显示"淘汰"），宣读时以"建议"列为准，跳过淘汰项即可。

---

## 3. 散会前 5 分钟

- 确认每个"待验证 / 进入 24h 验证"的项目都有：实验内容 + 负责人 + 24h 期限；
- 确认保留意见都已落到 X 列（口头分歧不留档等于没发生）;
- 保存文件。实验做完后更新证据索引，下次会议重评——**没有新证据，不允许靠辩论提分**。
