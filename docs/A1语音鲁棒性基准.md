# A1 语音鲁棒性自动基准

> 状态：v1 已实现；先跑协议校准，再按统计停止规则扩样。
>
> 目标不是消耗 Step Plan 额度，而是在无家庭素材、无需逐条人工标注的前提下，
> 给 Stepfun 云端音频能力与 Spark 本地 Step-Audio 2 mini 产出可复跑的同题 A/B 证据。

## 1. 裁判与被测模型分离

每条旁白先由确定性模板从结构化事实生成，事实包括物品、主人、旧位置、新位置、
打包组和颜色。这个结构化对象是唯一真值；模型只做两件事：

1. StepAudio 2.5 TTS 把旁白文本合成为音频；
2. 云端 StepAudio 2.5 Chat 或 Spark 本地 Step-Audio 2 mini 从音频抽取冻结的 A1 JSON。

模型不生成标准答案，也不参与自己的判卷。预测只进入验收报告，不自动写入产品契约、
真值或冻结配置。

## 2. 覆盖矩阵

v1 覆盖：

- 话术：直接叙述、信息重排、带口语停顿；
- 复杂度：每条 1～3 件物品；
- 信息完整度：全字段、主动省略一个可空字段；
- 音色：`cixingnansheng`、`linjiajiejie`；
- 音频条件：干净、20 dB/10 dB 白噪声、0.9× 语速、32 kbps 编解码回环；
- 指标：Schema 合法率、整条全对率、物品召回/精度、字段准确率。

正式 A/B 两端统一使用零-shot、温度 0 的贪心解码。温度 0.2 只保留在早期协议校准
记录中；不同解码策略的结果不得混为同一 A/B。

家庭影像和家庭历史语音不进入该基准。合成 WAV 只留在 gitignore 的 `local-data/`；
仓库只保存真值文本、预测判卷、哈希、模型版本、用量和统计摘要。

## 3. 统计停止规则

主指标为字段准确率，报告 95% Wilson 区间。默认目标是每个音频条件的区间半宽
不超过 0.08，并要求按最坏比例预先满足每条件最少样本数：

- 半宽 0.08：147 条/条件，5 条件共 735 个观测/后端；
- 半宽 0.05：381 条/条件，5 条件共 1905 个观测/后端。

即使早期结果接近 0% 或 100%，也不利用“看起来很稳”提前缩样；先满足最坏情形样本数。
`coverage_progress` 同时要求每个后端的已判卷观测数等于计划数；覆盖未满时，即使局部
区间看似很窄也不得触发停止。token/call 用量仅作预算护栏，不进入成功判定。
调用可断点续跑，每次以 `--max-new-tts` / `--max-new-extractions` 限制新增外部调用。

首批协议校准固定为 6 条旁白 × 3 条件 = 18 个观测/后端。它只回答“链路、扰动、
判卷和 A/B 是否可信”，不得宣称统计收敛。

## 4. 本地云端校准

```bash
python scripts/a1_robustness.py init \
  --run-dir local-data/a1/calibration-20260716-v1 \
  --base-cases 6 \
  --conditions clean,noise20,speed090

python scripts/a1_robustness.py cloud \
  --run-dir local-data/a1/calibration-20260716-v1 \
  --max-new-tts 6 \
  --max-new-extractions 18

python scripts/a1_robustness.py score \
  --run-dir local-data/a1/calibration-20260716-v1 \
  --report-dir results/acceptance/A1/calibration-20260716-v1
```

`cloud` 命令可重复执行：已有 TTS、扰动音频和预测按文件跳过，不重复计费。

## 5. Spark 原地云/本地 A/B

赛方 2026-07-16 通知尽量避免 SSH 文件传输并主动限流后，不再 rsync 合成音频。
本会话首次连接仍先执行安全门；模型加载前必须检查内存。Step-Audio 环境和模型权重
分别位于 `~/envs/stepaudio` 与 `~/models/stepfun-ai__Step-Audio-2-mini`。代码通过小体积
deploy 同步，计划、音频、两端预测都在 Spark 原地生成，只拉回最终判卷摘要。

```bash
./scripts/spark_healthcheck.sh
./scripts/deploy.sh

ssh spark 'free -h'

python scripts/a1_spark_factory.py launch \
  --acknowledge-key-exposure \
  --run-dir local-data/a1/calibration-20260716-v3-spark \
  --report-dir results/acceptance/A1/calibration-20260716-v3-spark \
  --base-cases 6 \
  --conditions clean,noise20,speed090 \
  --max-new-tts 6 \
  --max-new-extractions 18 \
  --max-new-local 18 \
  --revocation-delay-days 0 \
  --log logs/a1_factory_calibration_v3.log

ssh spark 'tail -n 50 ~/proj/logs/a1_factory_calibration_v3.log'
```

`launch` 只有在显式传入 `--acknowledge-key-exposure` 后才从本地 `.env` 读取一次性
key，并仅通过 SSH stdin 注入远端进程内存。key 不进
命令行、日志、状态文件或远端磁盘；云阶段结束即从环境移除，但仍必须从注入时起按
已泄露处理并在批次完成后撤销。工厂后台完成云端温度 0 抽取、本地温度 0 抽取和统一
判卷；最终仅拉回小体积 `results/acceptance/A1/<run_id>/`，不拉音频和原始预测池。

## 6. 扩样裁决

只有首批同时满足以下条件才生成默认 735 观测/后端的正式计划：

1. 两个后端都能稳定产出可解析 JSON；
2. 真值规范化没有系统性误判（例如位置同义表达被当错）；
3. 20 dB 噪声和语速变化确实生效，音频哈希不同；
4. 云端与本地结果的差异可追溯到模型输出，而不是 prompt、输入或判卷不一致；
5. 运行时间、内存和 Step Plan 用量在 manifest 中完整记账。

若校准暴露协议缺陷，先修协议并换新 `run_id` 重跑；旧结果保留为失败证据，不在原目录
覆盖。正式批次达到停止门即停止，不以额度剩余量追加样本。
