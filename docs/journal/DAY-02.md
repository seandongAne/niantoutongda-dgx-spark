## 失败与教训

- “七 Agent”存在角色膨胀风险；评分看协同质量与能力差，不看框的数量，因此收敛为四个异构主链 Agent加一个条件声学 Agent。
- 让第二 LLM 审计引用既增加概率性漏检，也模糊安全责任；最终裁决为确定性引用门唯一授权上屏，同模型额外推理只能是可选复核 pass。
- “解释流式上屏”与“审计通过才上屏”发生 P0 冲突；v0.4 改为确定性告警即时显示、生成草稿服务端缓冲、审计后整体显示。
- 文档版本名从 `_v0_4` 漂移到 `v0.4`，同时暴露互链残留问题；命名变更必须连同引用一起检查。
- TTS 若没有硬时间盒会吞噬主线，因此仅在 Step-Audio 2 mini 通过后执行，下载到首个 WAV 共用 30 分钟。
- 初版十日谈把 2026-07-12 直接算成 D1，导致日志整体偏移一天；正式十天赛程应为 D1=2026-07-13、D10=2026-07-22，准备日必须单列 D0。发现后立即修正手册、日志、汇编稿与 Agent 纪律。
- 原“72 小时主干 + D4–D10”排期描述的是执行工时，不代表当前已经完成 72 小时主干；在零切片 PASS 的真实基线上，必须把 G0/SP0、S1–S7 与评分切片重新映射到日历，并明确多人并行前提。
