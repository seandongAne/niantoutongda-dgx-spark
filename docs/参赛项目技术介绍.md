# AI 搬家复原 Agent：让 DGX Spark 记住一个家的秩序

搬家真正困难的部分，往往不是把物品从 A 点运到 B 点，而是把一个家已经形成的使用习惯、物品关系和空间秩序，一起迁移到新环境。围绕这个问题，我们实现了一套运行在 NVIDIA DGX Spark 上的多模态搬家复原系统：它从旧居视频和口述中建立物品记忆，在新居视频中理解可用空间，最后生成可以直接执行和验收的搬运方案。

整条链路接收三段旧居视频、一段口述和一段新居扫描。Step-Audio 2 mini 将自然语言整理成结构化生活线索；Grounding DINO 完成开放词汇目标发现，DINOv2 冻结视觉骨干配合轻量投影头提取跨视角特征，再通过匈牙利匹配、质量门和 ReID 约束，把同一件物品在不同镜头中的碎片重新连接起来。系统由此得到的不是一份简单的检测列表，而是一份带证据、关系和置信度的“家庭物品记忆”。

## 从物品记忆到可执行计划

主链由四个 Agent 协同完成。MEM Agent 维护物品身份、跨镜追踪和最小澄清问题；GROUP Agent 根据口述与视觉证据恢复“哪些东西应该一起搬”的生活组合；SPACE Agent 将新居视频转化为候选区域图，并调用 OR-Tools CP-SAT，在容量、支撑、电源、共置和互斥等约束下求解布局；EXEC Agent 则把结果编译成任务卡，调度验收并汇总最终状态。

Agent 之间不依赖隐式上下文，而是通过 Pydantic v2 定义的 JSON/JSONL 消息通信。每条消息都带有 `message_id`、因果引用、生产者和 SHA-256 载荷哈希，因此一次运行可以严格回放，任意结论也能一路追溯到原始视频、模型输出和约束求解结果。验收阶段已经在 Spark 上以独立进程完成 MEM/SPACE 并行 fan-out 与 EXEC 确定性 fan-in，让多 Agent 从概念分工真正落到可验证的进程边界。

## 在一台 Spark 上完成感知、推理与编排

系统以 DGX Spark 的 GB10 与 128 GB 统一内存为计算底座，将视觉、语音、视觉语言模型和组合优化收进同一台本地设备。Nemotron Nano 12B V2 VL 采用 NVFP4 权重并通过 vLLM 服务化，图像任务吞吐从约 1.7 tok/s 提升到 25.4 tok/s，约为原始 BF16 Transformers 路径的 15 倍。研发阶段再由 StepFun 3.7 Flash 对难样本进行错误归因和伪标签建议，Spark 本地训练 DINOv2 投影头并扫描 18 组工作点，使 ReID R@1 从 0.8031 提升到 0.8083，同时将澄清量从 1379 降至 677。

| 层级 | 核心技术 |
| --- | --- |
| 多模态感知 | Step-Audio 2 mini、Grounding DINO、DINOv2、OpenCV |
| 本地模型服务 | Nemotron Nano 12B V2 VL、NVFP4、vLLM |
| 身份与关系建模 | ReID、轻量投影头、匈牙利匹配、硬门控 |
| Agent 与求解 | Python、Pydantic v2、OR-Tools CP-SAT |
| 证据工程 | JSONL、YAML、SHA-256 内容寻址、严格 trace replay |

面向视觉推理链路的 TensorRT 部署已完成输出一致性收敛：在定位并以图级标记绕过一处融合精度缺陷后，FP32 引擎与 PyTorch 基线在冻结负载上检测决策一致（分数差 2.5e-4，p50 629.96 ms），FP16 引擎实测 2.49×（p50 273.15 ms）、其采纳待贴阈值稳定性口径落地；当前生产路径采用 torch.compile FP32（1.170×），全部测量已接入同一套可回放证据链。

## 已跑通的完整结果

在完整实测中，系统已将 **3306 个原始视觉实体收敛为 20 件可信物品**，并把需要确认的问题控制在 4 个以内；20 件物品被组织成 **3 个生活组合与 2 个独立装箱单元**，覆盖率为 20/20。新居侧从 **2278 条空间观测中形成 168 个候选区域**，最终自动确定 5 个放置区域，独立空间评分为 5/5。

布局求解输出 `PLAN_READY`，5 个搬运单元被编译为 5 张任务卡；四 Agent 主链严格回放结果为 `PASS`，最终交付包包含 44 份经过 SHA-256 校验的阶段产物。至此，旧居里的声音、物品和关系，被编译成了新居里可执行、可验收、可追溯的一组动作——一台 Spark 保存的不只是物品清单，而是一套可以被重新建立的生活秩序。
