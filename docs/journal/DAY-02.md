## 失败与教训

- 我觉得“七 Agent”存在角色膨胀风险；评分看协同质量与能力差，不看框的数量，因此我选择四个异构主链 Agent 加一个条件声学 Agent。
- 我觉得让第二 LLM 审计引用既会增加概率性漏检，也会模糊安全责任；我最后认为，只有确定性引用门可以授权上屏，同模型额外推理只能作为可选复核 pass。
- 我发现“解释流式上屏”与“审计通过才上屏”存在 P0 冲突；因此我在 v0.4 中改为确定性告警即时显示、生成草稿服务端缓冲、审计后整体显示。
- 我发现文档版本名从 `_v0_4` 漂移到 `v0.4`，同时还存在互链残留问题；我认为命名变更必须连同引用一起检查。
- 我觉得 TTS 若没有硬时间盒会吞噬主线，因此我只会在 Step-Audio 2 mini 通过后执行，并把从下载到生成首个 WAV 的全过程限制在 30 分钟内。
- 我在初版十日谈中把 2026-07-12 直接算成 D1，导致日志整体偏移一天；我认为正式十天赛程应为 D1=2026-07-13、D10=2026-07-22，准备日需要单列 D0。发现后，我立即修正了手册、日志、汇编稿与 Agent 纪律。
- 我原先写的“72 小时主干 + D4–D10”排期描述的是执行工时，不代表我已经完成 72 小时主干；在零切片 PASS 的真实基线上，我需要把 G0/SP0、S1–S7 与评分切片重新映射到日历，并明确多人并行前提。
